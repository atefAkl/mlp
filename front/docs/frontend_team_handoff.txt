FRONTEND TEAM HANDOFF - SUBSCRIPTIONS MODULE
Language: Arabic
Scope: Frontend integration guide with backend APIs
Note: This handoff is provided as TXT (non-MD) as requested.

==================================================
1) الهدف
==================================================
هذا الملف يوضح لفريق الفرونت كيف يربط الواجهات مع الباك اند الخاص بالاشتراكات:
- صفحة اشتراك المشترك (Tabs + Forms)
- صفحة متابعة الاشتراك للمشترك
- صفحة إدارة الأدمن للمشتركين
- صفحة إدارة البيانات المرجعية


==================================================
2) البيئة والاعتماديات
==================================================
Base API URL (local):
http://127.0.0.1:8000/api

Auth:
- Login endpoint يرجع token
- كل endpoints الأدمن تحتاج:
Authorization: Bearer {token}

Content-Type:
- JSON في معظم requests
- form-data عند رفع ملفات CV/CR

==================================================
A) صفحة المشترك

- Filters (type/status/name/date/round)

D) صفحة الأدمن - المرجعيات
- CRUD لقوائم:
countries, stacks, positions, rounds, company_fields, package_ranges


==================================================
4) API CONTRACTS المهمة للفرونت
==================================================

4.1 Public APIs

(1) GET /subscriptions/meta
Use case:
- تعبئة قوائم select

Response keys:
- countries
- stacks
- positions
- rounds
- company_fields
- package_ranges
- interview_times


(2) POST /subscriptions
Use case:
- إرسال نموذج الاشتراك

Request type:
- multipart/form-data

General fields:
- type
- email
- full_name
- country_id
- accept_conditions

Type-specific fields:
- trainee: stack_id, round_id, interview_times[], cv(optional)
- trainer: position_id, interview_times[], cv(required)
- company: brand_name, company_field_id, package_range_id, cr_number, extra_information(optional), cr(optional)

Success response (201):
- message
- subscription_id
- redirect_url

Validation error (422):
- errors object per field

Rate limit:
- 20 requests per minute per IP


(3) GET /subscriptions/public/{publicId}
Use case:
- صفحة متابعة الطلب

Response:
- subscription_id
- message
- subscription (structured object)


4.2 Admin APIs (auth required)

(1) GET /admin/subscribers
Use case:
- القائمة الأساسية + pagination

Filters:
- type
- status
- name
- date
- round_id
- per_page


(2) GET /admin/subscribers/grouped
Use case:
- عرض مجمع حسب النوع للـ cards/sections

Optional params:
- type
- status
- name
- date
- round_id
- per_type_limit

Response:
- filters
- groups.trainee.total/items
- groups.trainer.total/items
- groups.company.total/items


(3) POST /admin/subscribers/selection
Use case:
- helper للـ bulk selection في الواجهة

Request:
- action: select_all | select_none | invert
- selected_ids[] (optional)
- نفس filters القائمة

Response:
- matching_count
- selected_count
- selected_ids[]


(4) GET /admin/subscribers/stats
Response:
- total_subscribers
- totals_by_type
- totals_by_status
- daily_counts_last_7_days


(5) GET /admin/subscribers/trends
Params:
- interval: daily|weekly
- days: 1..90
- type (optional)
- status (optional)

Response:
- interval
- from_date
- to_date
- filters
- data[] buckets


(6) GET /admin/subscribers/{type}/{id}
Use case:
- تفاصيل المشترك


(7) PATCH /admin/subscribers/{id}/status
Request:
- status: accepted|rejected|maybe
- scheduled_at (required for accepted)
- reason (required for rejected/maybe)


(8) PATCH /admin/subscribers/bulk-status
Request:
- ids[]
- status: rejected|maybe
- reason


(9) POST /admin/subscribers/{id}/resend-email
Request (optional):
- event_type: submitted|accepted|rejected|maybe
- reason
- scheduled_at


4.3 Admin Reference Data APIs

(1) GET /admin/reference-data
(2) POST /admin/reference-data/{resource}
(3) PUT /admin/reference-data/{resource}/{id}
(4) DELETE /admin/reference-data/{resource}/{id}

Allowed resource values:
- countries
- stacks
- positions
- rounds
- company_fields
- package_ranges

6) سيناريوهات UX موصى بها
==================================================

A) Subscription Create UX
1. عند تحميل الصفحة:
- call /subscriptions/meta
- cache options في state

2. عند تغيير tab:
- reset type-specific fields فقط
- احتفظ بالحقول العامة

3. قبل submit:
- client-side required validation
- disable submit أثناء الإرسال

4. بعد success:
- toast success
- redirect إلى route الملخص

B) Admin Status UX
1. Accept button:
- open modal with datetime picker
- send PATCH status=accepted + scheduled_at

2. Reject/Maybe:
- open modal with reason textarea
- send PATCH status + reason

3. بعد نجاح التحديث:
- refresh row/card
- refresh stats optionally

C) Bulk UX
1. Use /selection endpoint for select_all/invert correctness with filters
2. Enable bulk action buttons only when selected_count > 0
3. Confirm modal before bulk update

7) أخطاء متوقعة وكيف يتعامل معها الفرونت
==================================================
- 401 Unauthorized:
  token مفقود/منتهي -> redirect login

- 403 Forbidden:
  user لا يملك صلاحية إدارة -> show permission message

- 404 Not found:
  id/public_id غير صحيح -> show not found view

- 422 Validation:
  اربط errors[field] مع inputs مباشرة

- 429 Too Many Requests (create endpoint):
  اعرض رسالة "تم تجاوز الحد المسموح، حاول بعد قليل"

- 500:
  generic error toast + retry option
